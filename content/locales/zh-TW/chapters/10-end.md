# 0 未來傳送門：BF系列創意的未來

> ――以變化為前提創造的牢不可破的思維

* 我們做事的方式發生了變化：隨著工具規格、AI 行為和 UI 標準的變化，最佳實踐也會改變。如果你把目前的“正確答案”記為“固定正確答案”，那麼每次有變化你就必須重新做一遍。
* 改變是一個機會：規則編輯器和地圖編輯器的功能越多，就越有可能用更少的程式碼使它們更安全。
* 當您學會如何關注事物時，焦慮就會消失：什麼可能會發生變化，以及如果事物發生變化應在哪裡進行審查 - 如果您有監控點和更換程序，那麼更新就不會令人恐懼。

> 本章並不是試圖預測未來，而是總結思維方式，以避免在未來到來時迷失方向。

# 1 容易改變的事情和難以改變的事情

首先，要組織起來。容易改變=每次都回顧；難以改變=長期重複使用。

## 可變的

* UI禮儀（如何顯示訊息、HUD標準）
*「如何玩」SFX/FX（建議冷靜和擁擠措施）
* XP的處理和發布（根據情況隨時注意限制的可能性）

## 很難改變

* **「文字→地標→效果」的順序**（基本溝通方式）
* 按名稱管理 ID (ids.ts)
* 指定狀態並建立只能走一次的路徑（Phase和onceIn）
* 三個盒子的邊界/狀態/呈現（第7章分區設計）

> 適應未來是一種「方法」。我們繼續保護的是我們的「思維方式」。
> 一旦你能做出這種區分，改變就會從「麻煩」變成「有趣的調整」。

# 2 順應工具進化的“重讀”
## 2 .1 規則編輯器數量增加/變化

* 取代：當新事件和新動作增加時，不要直接到處呼叫它們，而是先在 `api.ts` 或 `ui.ts` 中建立自己的呼叫函數。不要增加可直接從 `main` 或 `Script.ts` 存取的位置數量。
* 檢查：是否與現有的onceIn（單通道）相容？更換為防止多次發射的結構。

在程式設計中，這種精簡的呼叫函數有時被稱為「包裝器」。
這裡重要的不是術語，而是即使 Portal API 的名稱發生變化，也有一個地方可以修復它。

## 2 .2 地圖編輯器中的物件增加

* 解釋：依照第四章的規則，只是不要重複ID/增加系列數。
* 檢查：注意共享系統的底圖相容性。只需「像往常一樣」查看是否出現警告標記（遵循第 4 章中的習慣）。

## 2 .3 UI 標準的變更（通知、圖釘、圖示）

* 替換：ui.say 現在可以在一個地方替換（第 7 章和第 8 章）。
* 檢查：不要違反「一次只做一件事」的原則。即使出現新的 UI，也不要同時使用多個。
*效果：即使製作「吵鬧」的標準隨著時間的推移而改變，如果設計不是太吵鬧，它也會保持不變。

# 3 AI與腳本的關係（為未來的成長做好準備）

人工智慧是一個進步將極大受益的領域。

* 目前的做法：使用AI_Spawner和AI_WaypointPath來確保彈簧前面的空間和鬆散的角落（第4章）。
* 成長潛力：如果AI行為API數量增加，將分為**``動作是數據，啟動訊號是事件。 ''**
  * 範例）將**操作配方**定義為固定值，並僅在規則中稱為**“啟動配方A”**。
* 原因：AI的詳細指令很容易改變。 **訊號的骨架（推動→引導→到達→表現）**很難改變。
*實施說明：從現在開始，透過將AI相關程序作為一組程序儲存在專用位置，將來替換AI相關程序將更容易。

# 4 性能與規模（主動習慣）

未來，當更大的地圖和更密集的戰鬥成為常態時，擁有重量級的構建將會派上用場。

* 避免處理每一幀（第 8、9 章）：每 0.5 到 1 秒更新一次距離和引導就足夠了。
* 自行決定同時播放的上限（例如，SFX/FX 的「上限 3」）。
*“專為切割而設計”：不會在長距離產生效果。向 ui 新增距離過濾器。
* 測試習慣：插入一個完整測試的操作（第9章）。
  * → 可以使用相同的程序來驗證未來的「容易變得困難的場景」。

# 5 發表後如何在小組中成長

模式只有在玩的過程中才會成長。
然而，在Portal中，僅僅透過自然流動進入匿名個人模式來聚集人群並不容易。
如果沒有來自主要分銷商或現有社區的引導線，則更安全的做法是不要設計太多，並假設大量人員將繼續聚集。

本書本身並不是為了吸引顧客。
發布之後我們需要看到的是一小部分人是否能夠毫不猶豫地開始，是否無法進步，以及是否能夠弄清楚下一步要修復哪裡。

* 養成256個或更少字元的短句模板（第9章）的習慣：目的/人數/要求/第一個行動。
* 不要假設會補充標籤：假設沒有創建者可以自由添加的標籤，請在標題、描述、縮圖和外部公告中傳達這一點。
* 小改進→立即發布：根據少數人的印象，修正不清楚的地方或過長的場景。
* 不要強迫外部公告：選擇一個可以輕鬆繼續發文的地方，例如 X、Discord、部落格、Note 等。

# 6 「未來阻力」清單（每六個月）

每六個月檢查一次以下內容就足夠了。

1. ids.ts：使用 Vitest 檢查 -1、重複和未知 ID。使用 ObjIdManager 或第 4 章中的帳本檢查 Godot 端的放置 ID。
2. ui.ts：說/引導/慶祝的順序/頻率（不吵鬧）。
3. config.ts：防禦秒數/冷卻時間/建議玩家數量的數值適合目前環境。
4.流程（轉換表）：沒有太多的分支（不需要的分支被折疊）。
5.效能：滿載只進行一次負載測試。
6. 說明：不脫離實際。了解256個字以內的目的、人數、要求、第一個行動。

# 7 將“今天的正確答案”變成“明天的材料”

如果只是製作、出版然後就到此為止，那就太浪費了。

* 重複使用：將 ui、api、game 和 ids 4 部分複製並貼上到下一個模式。
* 從小處開始：當新功能出現時，小範圍地嘗試並將其合併到模板中。
* 放棄的勇氣：在下一個模式中，你不能使用從第一步開始就變得太重的技巧。過去是圖書館，現在是野外遺址。

# 8 創造力的未來（創作者的觀點）

*「正確恐懼」保護創造力：以改變為前提，只固化牢不可破的核心（ID、地位、秩序）。
*「最小循環」成為一種文化：如果你能分享推動→引導→到達→引導的流程，多人可以快速創造它。
*「視覺化」帶來發明：當數值和狀態在調試HUD上可見時，實驗→學習→改進的速度就會加快。

> 未來不僅僅是「更多新武器」。
> 「溝通規則」將會更新。你只需要每次**稍微**重寫它。

# 結論

* 第10章不是預測未來的章節，而是確立如何製造不會被未來妥協的東西的章節。
* 保護的是ID命名/狀態和單段/文字→地標→效果/三格劃分。
* 變化的是API的呼叫方式、UI禮儀以及詳細的AI動作。您所要做的就是修復 API 或 UI 的一部分。
* 透過半年一次的未來抵抗檢定來延長模式的壽命。
