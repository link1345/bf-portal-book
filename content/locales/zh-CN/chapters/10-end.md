#0 未来传送门：BF系列创意的未来

> ――以变化为前提创造的牢不可破的思维

* 我们做事的方式发生了变化：随着工具规范、AI 行为和 UI 标准的变化，最佳实践也会发生变化。如果你把当前的“正确答案”记为“固定正确答案”，那么每次有变化你就必须重新做一遍。
* 改变是一个机会：规则编辑器和地图编辑器的功能越多，就越有可能用更少的代码使它们更安全。
* 当您学会如何关注事物时，焦虑就会消失：什么可能会发生变化，以及如果事物发生变化应在哪里进行审查 - 如果您有监控点和更换程序，那么更新就不会令人恐惧。

> 本章并不是试图预测未来，而是总结思维方式，以避免在未来到来时迷失方向。

#1 容易改变的事情和难以改变的事情

首先，要组织起来。容易改变=每次都回顾；难以改变=长期重复使用。

## 可变的

* UI礼仪（如何显示消息、HUD标准）
*“如何玩”SFX/FX（建议冷静和拥堵措施）
* XP的处理和发布（根据情况随时注意限制的可能性）

## 很难改变

* **“文字→地标→效果”的顺序**（基本沟通方式）
* 按名称管理 ID (ids.ts)
* 指定状态并创建只能走一次的路径（Phase和onceIn）
* 三个盒子的边界/状态/呈现（第7章分区设计）

> 适应未来是一种“方法”。我们继续保护的是我们的“思维方式”。
> 一旦你能做出这种区分，改变就会从“麻烦”变成“有趣的调整”。

#2 顺应工具进化的“重读”
## 2.1 规则编辑器数量增加/变化

* 替换：当新事件和新动作增加时，不要直接到处调用它们，而是先在 `api.ts` 或 `ui.ts` 中创建自己的调用函数。不要增加可直接从 `main` 或 `Script.ts` 访问的位置数量。
* 检查：是否与现有的onceIn（单通道）兼容？更换为防止多次发射的结构。

在编程中，这种精简的调用函数有时被称为“包装器”。
这里重要的不是术语，而是即使 Portal API 的名称发生变化，也有一个地方可以修复它。

## 2.2 地图编辑器中的对象增加

* 解释：按照第四章的规则，只是不要重复ID/增加系列数。
* 检查：注意共享系统的底图兼容性。只需“像往常一样”查看是否出现警告标记（遵循第 4 章中的习惯）。

## 2.3 UI 标准的变化（通知、图钉、图标）

* 替换：ui.say 现在可以在一处替换（第 7 章和第 8 章）。
* 检查：不要违反“一次只做一件事”的原则。即使出现新的 UI，也不要同时使用多个。
*效果：即使制作“吵闹”的标准随着时间的推移而改变，如果设计不是太吵闹，它也会保持不变。

#3 AI与脚本的关系（为未来的成长做好准备）

人工智能是一个进步将极大受益的领域。

* 目前的做法：使用AI_Spawner和AI_WaypointPath来确保弹簧前面的空间和松散的角落（第4章）。
* 增长潜力：如果AI行为API数量增加，将分为**``动作是数据，启动信号是事件。''**
  * 示例）将**操作配方**定义为固定值，并仅在规则中将其称为**“启动配方A”**。
* 原因：AI的详细指令很容易改变。 **信号的骨架（推动→引导→到达→表现）**很难改变。
*实施说明：从现在开始，通过将AI相关程序作为一组程序存储在专用位置，将来替换AI相关程序将更容易。

# 4 性能和规模（主动习惯）

未来，当更大的地图和更密集的战斗成为常态时，拥有重量级的构建将会派上用场。

* 避免处理每一帧（第 8、9 章）：每 0.5 到 1 秒更新一次距离和引导就足够了。
* 自行确定同时播放的上限（例如，SFX/FX 的“上限 3”）。
*“专为切割而设计”：不会在长距离产生效果。向 ui 添加距离过滤器。
* 测试习惯：插入一个完整测试的操作（第9章）。
  * → 可以使用相同的程序来验证未来的“容易变得困难的场景”。

# 5 发表后如何在小组中成长

模式只有在玩的过程中才会成长。
然而，在Portal中，仅仅通过自然流动进入匿名个人模式来聚集人群并不容易。
如果没有来自主要分销商或现有社区的引导线，则更安全的做法是不要设计太多，并假设大量人员将继续聚集。

本书本身并不是为了吸引顾客。
发布之后我们需要看到的是一小部分人是否能够毫不犹豫地开始，是否无法进步，以及是否能够弄清楚下一步要修复哪里。

* 养成256个或更少字符的短句模板（第9章）的习惯：目的/人数/要求/第一个行动。
* 不要假设会补充标签：假设没有创建者可以自由添加的标签，请在标题、描述、缩略图和外部公告中传达这一点。
* 小改进→立即发布：根据少数人的印象，修正不清楚的地方或过长的场景。
* 不要强迫外部公告：选择一个可以轻松继续发帖的地方，例如 X、Discord、博客、Note 等。

# 6“未来阻力”清单（每六个月）

每六个月检查一次以下内容就足够了。

1. ids.ts：使用 Vitest 检查 -1、重复和未知 ID。使用 ObjIdManager 或第 4 章中的账本检查 Godot 端的放置 ID。
2. ui.ts：说/引导/庆祝的顺序/频率（不吵闹）。
3. config.ts：防御秒数/冷却时间/推荐玩家数量的值适合当前环境。
4.流程（转换表）：没有太多的分支（不需要的分支被折叠）。
5、性能：满负荷只进行一次负载测试。
6. 说明：不脱离实际。了解256个字以内的目的、人数、要求、第一个行动。

#7 将“今天的正确答案”变成“明天的材料”

如果只是制作、出版然后就到此为止，那就太浪费了。

* 重用：将 ui、api、game 和 ids 4 部分作为模板复制并粘贴到下一个模式。
* 从小处开始：当新功能出现时，小范围地尝试并将其合并到模板中。
* 放弃的勇气：在下一个模式中，你不能使用从第一步开始就变得太重的技巧。过去是图书馆，现在是野外遗址。

#8 创造力的未来（创作者的视角）

*“正确恐惧”保护创造力：以改变为前提，只固化牢不可破的核心（ID、地位、秩序）。
*“最小循环”成为一种文化：如果你能共享推动→引导→到达→引导的流程，多人可以快速创建它。
*“可视化”带来发明：当数值和状态在调试HUD上可见时，实验→学习→改进的速度就会加快。

> 未来不仅仅是“更多新武器”。
> “沟通规则”将更新。你只需要每次**稍微**重写它。

# 结论

* 第10章不是预测未来的章节，而是确立如何制造不会被未来妥协的东西的章节。
* 保护的是ID命名/状态和单段/文字→地标→效果/三格划分。
* 变化的是API的调用方式、UI礼仪以及详细的AI动作。您所要做的就是修复 API 或 UI 的一部分。
* 通过半年一次的未来抵抗检查来延长模式的寿命。
